# Changes

Chronological record of notable changes to Clearance. The founding documentation phase predates this file; entries begin where durable decisions and structure land.

## 2026-05-27

- Settled the v1 founding decisions (OB-881): Apache 2.0 license, npm-workspaces monorepo, `@clearance/*` package scope, Postgres-only storage for v1, reserved system tags plus a documented custom namespace, MCP-first interface with REST/dashboard out of v1, and an optional (never required) review bot. Added `docs/architecture/decision-records.md`, marked the founding plan's decisions as settled, and noted the license in the README.
- Implemented the `@clearance/mcp` write tools and verified close (OB-887): the mutating tool set on the same server — `create_candidate`, `create_work_item`, `update_work_item`, `claim_next` (race-safe via `FOR UPDATE SKIP LOCKED`, expires stale claims, one active claim per item), `claim_heartbeat`, `block_with_child` (creates the blocker child, blocks + demotes the parent, links them, releases the claim), `close_verified` (validates evidence for tests/smoke/deploy/docs/migration/no_code), and `run_pack_create` / `run_pack_record`. Enforces the governance model: agents cannot self-apply `autonomous_safe` (requires `operator_grant`), unknown tags are rejected, and run packs never grant safety; all mutations are transaction-safe and governance failures return `isError` tool results. 10 happy-path + guardrail tests; updated the worker-execution, first-work-item, candidate-review, and digest-review guides.
- Implemented the `@clearance/mcp` read tools and operational digest (OB-886): an MCP server (stdio, `clearance-mcp` bin) exposing read-only `list_work_items` (status/tag/work_type/category filters), `get_work_item`, `list_work_item_candidates`, `list_project_tracks`, `list_claims`, `list_run_packs`, and `digest` (ready queue / blocked / active+stale claims / pending candidates / recent completions). Bounded output (default 50, max 200) with pagination, parameterized queries, and a stable JSON shape. Tested with fixtures against a real database, including a client↔server round-trip over the SDK's in-memory transport. Wired `@clearance/schema` as a workspace dependency (tsconfig paths + vitest alias resolve it to source).
- Added `@clearance/schema` types, the `clearance` migrate CLI, and install validation (OB-885): hand-maintained TypeScript types for every table and controlled vocabulary (exported from the package), a `clearance` bin with `migrate` (apply pending) and `status` (show applied/pending) commands that read `DATABASE_URL` and fail with clear messages, a `migrationStatus` helper, and a `migrationStatus` integration test. Updated `docs/guides/first-setup.md` with the real, runnable schema-setup flow.
- Implemented the `@clearance/schema` migrations and seed vocabulary (OB-884): a numbered Postgres migration set (`0001`–`0007`) that builds the full PMO substrate from an empty database — work items (with the governance CHECK invariants), candidates, governance tags, work-item tags/surfaces/links, machine-checkable preconditions, claims (one active lease per item), project tracks/links/candidates, run packs/items, and decision threads — seeded with the 8 reserved governance tags, 14 work types, 8 default surfaces, and 11 link relationships. Added an `applyMigrations` runner (per-file transactions, tracked in `clearance_migrations`, idempotent) and a fresh-database integration test (apply + idempotency + invariant enforcement) that runs against a throwaway DB and skips cleanly without one.
- Mapped the Nexus PMO source system into a generic extraction plan (OB-882): added `docs/reference/extraction-map.md` — the table-by-table lineage matrix (15 core tables + decision threads + vocab), per-column keep/rename/genericize/drop decisions grounded in live schema inspection, the controlled vocabularies (8 reserved tags, 14 work types, link/relationship sets, status enums), the governance invariants to preserve (the two `work_items` CHECK constraints + the no-self-promotion rule), Nexus-specific behavior not extracted, and the open questions for schema implementation.
- Scaffolded the public repo workspace and baseline tooling (OB-883): npm-workspaces monorepo with placeholder `@clearance/schema`, `@clearance/mcp`, and `@clearance/contracts` packages (no runtime behavior yet); baseline checks via `npm run check` (Prettier format, ESLint, TypeScript typecheck, Vitest, and an offline relative-Markdown-link check in `scripts/check-links.mjs`); the Apache 2.0 `LICENSE`; `CONTRIBUTING.md`; and `.editorconfig`/`.prettierrc`/`.prettierignore` hygiene. A fresh clone runs `npm install && npm run check`.
