# Documentation Standards

Clearance documentation should be clear enough for someone adopting the system without local context.

## Standards

- Lead with the PMO/work-management concept before implementation details.
- Define terms before using them heavily.
- Separate current behavior from planned behavior.
- Include concrete examples for abstract concepts.
- Link conceptual docs, guides, and reference pages together.
- Update docs in the same change that updates behavior.
- Avoid Nexus-only assumptions in public docs.
- Use Nexus history as provenance, not as the product headline.

## Required Doc Updates

When implementation lands, update:

- the relevant concept page
- the relevant guide
- the relevant reference page
- the README or docs index if navigation changes

## Tone

Use plain operational language. Clearance should feel like a practical PMO system a person can run, not a research paper or a prompt-only convention.
