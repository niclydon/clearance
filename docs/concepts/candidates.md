# Candidates

Candidates are proposed work or proposed projects waiting for review.

They exist because live systems create more possible work than should become real backlog. Humans, services, and agents can all discover plausible work. Clearance gives that discovery a review lane.

## Candidate-First Doctrine

Agents and automation should file candidates rather than directly creating approved work, except for concrete blocker children found while working a claimed item.

This keeps the PMO from filling with speculative, duplicate, low-context, or low-value tasks.

## Candidate Outcomes

A candidate can be:

- promoted into a real work item or project track
- rejected
- merged into existing work
- parked for later
- converted into a decision request

## Project Candidates

A project candidate proposes a larger tracked effort, such as a new repo, system integration, operating program, or major cleanup.

Project candidates should include enough evidence for a reviewer to understand why the project exists and what the first concrete action should be.

## Work Item Candidates

A work item candidate proposes a specific task.

It should include the proposed title, body, category, priority, source, and any relevant tags. Promotion can refine those fields before the item becomes executable.

## Current Status

Candidate creation and review are planned public features. The source PMO already uses candidate-first review, and Clearance will preserve it as a core behavior.
